# IFRIT — Monitoring Dashboard Design Context

> AI-Powered Fire Detection Monitoring Dashboard
> Design context for the impeccable skill.

---

## Design Context

### Users

**Primary operators:** Facility security operators sitting at a dedicated monitoring station in a security room, and building managers checking remotely from office desktops, laptops, or occasionally mobile devices between meetings.

**Tech literacy:** Moderate — these are operational professionals who understand building systems, sensor readings, and alert severity, but they are not software engineers. The interface must be immediately legible without training. Every data point should be self-explanatory.

**Context of use:** Dual-context. Security operators use this on a dedicated screen in a control room — often in dim lighting, for extended periods, watching for anomalies. Building managers check in periodically from brighter office environments to verify system health. Both contexts must be served.

**Job to be done:** Monitor real-time fire risk across a facility's rooms and sensors. Understand system health at a glance. Drill into individual rooms for sensor-level detail. Manage devices (cameras, MCU nodes). Respond to alerts with appropriate urgency. WhatsApp is the primary notification channel; the dashboard provides persistent visibility, historical data, and device management.

**Customer-facing:** This dashboard is the product. Clients log in to monitor their own facility. The UI must communicate professionalism, reliability, and trustworthiness — it IS the product experience.

### Brand Personality

**Voice:** Precise, confident, professional. Speaks with authority but never overwhelms. Data-forward — lets numbers and status speak for themselves.

**3-word personality:** Reliable · Precise · Vigilant

**Emotional goals:**
- Default state: "Everything is under control — the system is watching, I trust it." Calm confidence.
- Alert state: "Something needs attention — I know exactly what, where, and how serious." Clear urgency without panic.
- Overall: "This is a professional-grade tool that takes fire safety seriously." Not a toy, not a prototype.

**Brand name:** IFRIT — the dashboard is the product arm of the IFRIT platform. The brand should match the landing page identity: deep red, authoritative, controlled fire energy.

### Aesthetic Direction

**Visual tone:** Refined industrial — the precision of a Tesla car dashboard meets the data density of Grafana, filtered through the professionalism of a modern building management system. Clean surfaces, clear hierarchy, purposeful use of color for status, and restrained use of decoration.

**Primary palette:**
- **Red** as the primary brand accent — matching the IFRIT landing page. Deep, controlled ember red (not neon, not cherry). Used for the brand mark, critical alerts, and primary CTAs. In the dashboard context, red also carries semantic meaning (fire/danger), so use it with precision — never as decoration.
- **Amber/Orange** for warnings and elevated status. Already established in the sensor vocabulary.
- **Green** for safe/nominal status only.
- **Neutrals** — warm-tinted darks for dark mode (matching the landing page's dark sections), cool-shifted lighter neutrals for light mode. Never pure black or pure white. All neutrals tinted subtly toward the red-amber brand hue.

**Theme:** Both light and dark mode, with dark as the recommended default for monitoring stations. Light mode for building managers in office environments. Both must be fully designed — not a dark theme with a "just invert it" light mode.

**Typography direction:**
- A professional sans-serif for body and UI (NOT Inter — find something with the same clarity but more character).
- A monospace for data readouts, sensor values, and timestamps (system-critical information where tabular alignment matters).
- Fixed `rem` scales for the dashboard (no fluid type — this is a product UI, not a marketing page).

**Motion:** Minimal and purposeful. Sensor data should update smoothly (not jarring). Alert indicators pulse to draw attention. Page transitions are instant or very fast fade. No cinematic entrances — this is a work tool, not a showcase. Respect `prefers-reduced-motion`.

**References (conceptual):**
- Tesla's in-car UI: clean surfaces, clear hierarchy, information density without clutter
- Grafana's data presentation: serious data visualization that respects operator intelligence
- Apple's Home app: room-based organization, status at a glance, drilldown for detail
- Modern building management systems (Honeywell, Johnson Controls): professional trust

**Anti-references:**
- Sci-fi movie control rooms with gratuitous animations
- AI slop dashboards: glassmorphism, gradient buttons, glow effects, cyan-on-dark
- Over-designed admin templates with too many card types and decorative elements
- Military/defense aesthetic taken too far — this should be approachable, not intimidating

### Content Structure (Dashboard)

1. **Dashboard (Home)** — System-level overview. Total monitored rooms, device health, active alerts, key metrics. Telemetry chart showing recent sensor trends. Recent alert log. Device fleet status.
2. **Rooms** — All monitored rooms with live status. Direct status indicators per room (not grouped by status category). Drill into room detail.
3. **Room Detail** — Full sensor readout for a specific room: temperature, humidity, gas levels, camera feed link. Historical charts. Alert history for that room.
4. **CCTV Monitor** — Live camera feeds grid. AI detection overlay status.
5. **Devices** — MCU and camera inventory. Online/offline status, assignment to rooms. Should show human-readable room names, not raw UUIDs.
6. **Alerts** — Full alert history with severity, timestamps, acknowledgment status. Filterable and searchable.
7. **Notifications** — Notification settings (WhatsApp config, alert preferences).

### Design Principles

1. **Status at a glance.** The most important information — "is anything on fire?" — must be answerable in under 1 second from any page. Global status indicators, persistent alert badges, color-coded room states.

2. **Data density without clutter.** Security operators need to see a lot of information simultaneously. Pack information tightly, but use spacing, typography, and color hierarchy to keep it scannable. Think "dense but organized," not "minimal for minimalism's sake."

3. **Color means something.** Red = fire/critical. Amber = warning/elevated. Green = safe/nominal. Blue = informational. These semantic associations are non-negotiable in a safety-critical system. Never use status colors decoratively.

4. **Professional trust.** This is a product that protects lives and property. Every pixel must communicate reliability. No playful elements, no gratuitous animations, no visual gimmicks. The quality of the UI IS the trust signal.

5. **Consistent with IFRIT brand.** The dashboard is part of the IFRIT product family. Red brand accent, same typographic confidence as the landing page, same sense of controlled authority. The landing page sells the product; the dashboard IS the product.

### Accessibility

- WCAG AA contrast minimum (especially critical for status colors on light/dark backgrounds)
- `prefers-reduced-motion` respected — pulse animations and transitions degrade gracefully
- Semantic HTML throughout — tables for tabular data, proper heading hierarchy
- Focus-visible styles for keyboard navigation
- Screen reader labels for all status indicators and interactive elements
- Color is never the only indicator — always pair with text labels or icons
